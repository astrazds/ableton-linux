# Crash/hang during audio device enumeration

## Symptoms

Live dies right after `Audio In Out: Constructor finished`, the endpoint
enumeration boundary. Two unrelated faults land on this line: a crash during
init, and a hang (main thread parked, no window).

## Research

Crash: the prefix's `MMDevices\Audio` registry accumulates endpoint names
nested one level deeper per boot per absent device (`Speakers (Speakers
(…))`, 70+ deep observed) until enumeration blows up. mmdevapi's
`MMDevice_Create()` rebuilds the FriendlyName as `"<FormFactor> (<drv_id>)"`
and re-saves it on every call, including registry reloads whose stored
`drv_id` is already a finished FriendlyName. Present devices get a fresh flat
description from the driver each boot; absent ones (Bluetooth, unplugged USB,
monitor sources) only ever reload from the registry and nest forever. The
audio server reports flat names throughout; the nesting is generated by Wine
on read-back.

Hang: with WirePlumber dead, PipeWire publishes no real cards (only
`auto_null`) and winepulse blocks on the pulse socket at the same log line.

## Mitigations

- [../patches/0021-mmdevapi-stop-re-wrapping-reloaded-endpoint-Friendly.patch](../patches/0021-mmdevapi-stop-re-wrapping-reloaded-endpoint-Friendly.patch):
  `MMDevice_Create()` gains an `init_props` flag; name properties are written
  only for raw driver-supplied names, and registry reloads preserve stored
  properties.
- Already-corrupt prefix, once:
  `wine reg delete 'HKLM\Software\Microsoft\Windows\CurrentVersion\MMDevices\Audio' /f`.
  Active endpoints regenerate flat; absent devices are not recreated.
- Hang: restart `wireplumber.service`.

## Caveats

- Both faults end the boot log identically. Check
  `systemctl --user is-active wireplumber` and `pactl list cards short`
  (empty = degraded graph = the hang) before touching the registry.
- The patch stops the growth; existing corruption still needs the one-time
  clear.
